Detailed Target Architecture
Shared Email Integration Layer for AP and Procurement
Prepared for implementation planning
Architecture scope
Shared omnichannel email capability covering inbound ingestion, attachment intake, thread handling, outbound communication, routing, and governed agent access across AP and procurement.
Design principle
Email is a shared platform channel. Business execution remains domain-specific: AP continues on the document/case/reconciliation path and procurement continues on the request/quotation/analysis path.
Architecture basis: This design is aligned to the current platform structure where AP uses the shared document and case pipeline, procurement uses its own request and analysis hierarchy, and both domains rely on the same tenant, audit, and async foundations.
1. Architecture goal
Create a shared email capability that supports inbound email ingestion, attachment-based document intake, thread-based follow-up handling, outbound templated communication, agent-safe email access, routing into either AP or procurement workflows, and full audit, trace, and RBAC enforcement.
Core principle: Email is a shared platform channel. Business processing stays domain-specific. The email layer should stop at normalization, classification, routing, linking, and governed actions, then hand off to AP or procurement handlers.
2. Target high-level architecture
External mail providers such as Microsoft 365 or Gmail connect into a new platform app, apps.email_integration. That app contains four layers: mailbox connectivity, email normalization, email intelligence, and domain routing.
The domain routing layer hands work to AP or procurement handlers. AP continues through DocumentUpload, Invoice, APCase, ReconciliationResult, AgentOrchestrator, and SupervisorAgent. Procurement continues through DocumentUpload, SupplierQuotation, ProcurementRequest, AnalysisRun, ProcurementAgentOrchestrator, and procurement-specific agents.
3. New app structure
Recommended new app: apps.email_integration.
Suggested structure: admin.py, apps.py, models.py, urls.py, views.py, api/ for serializers and viewsets, services/ for mailbox, ingestion, routing, classification, thread linking, entity linking, approval, triage, policy, and provider adapters, domain_handlers/ for AP and procurement handlers, tools/ for scoped email tools, tasks.py, signals.py, enums.py, and migrations/.
Optional supporting app: apps.communications, only if you want a reusable layer for templates, notification policies, and delivery abstractions. Mailbox, thread, and routing logic should remain in email_integration.
4. Data model design
MailboxConfig: tenant-scoped mailbox registration and authentication metadata. Key fields include tenant, provider, mailbox address, mailbox type, auth mode, inbound/outbound flags, webhook and polling settings, sync health, default route, allowed sender domains, config_json, and is_active.
EmailThread: canonical conversation object. Tracks provider thread identifiers, normalized subject, first and last message timestamps, domain context, link status, and nullable links to APCase, ProcurementRequest, or SupplierQuotation.
EmailMessage: canonical inbound/outbound record. Stores provider IDs, headers, body text/HTML, direction, classification, trust level, intent, routing status, matched entity, raw headers, provider payload, trace_id, and optional links to created documents.
EmailAttachment: tracks every attachment independently, including content type, size, hash, blob path, scan/safety status, classification, linked DocumentUpload, and attachment processing status.
EmailParticipant: optional but useful for analytics and trust scoring. Stores email, display name, role type, trust level, linked vendor, and linked user.
EmailRoutingDecision: persists the routing outcome and evidence, including target domain, handler, entity, confidence, deterministic flag, rule name/version, LLM usage, reasoning summary, and final status.
EmailAction: captures every mutation taken because of an email, including action type, actor, target entity, payload, result, error message, and trace_id.
EmailTemplate: reusable outbound templates with domain scope, subject/body templates, required variables, approval requirements, and active status.
5. Important enum design
EmailMessageClassification should cover AP_INVOICE, AP_SUPPORTING_DOCUMENT, PROCUREMENT_QUOTATION, PROCUREMENT_PROPOSAL, PROCUREMENT_CLARIFICATION, APPROVAL_RESPONSE, INTERNAL_REVIEW_REPLY, GENERAL_QUERY, and UNKNOWN.
EmailProcessingStatus should cover RECEIVED, NORMALIZED, ATTACHMENTS_STORED, CLASSIFIED, LINKED, ROUTED, PROCESSED, FAILED, and IGNORED.
SenderTrustLevel should cover TRUSTED_INTERNAL, TRUSTED_VENDOR, KNOWN_EXTERNAL, UNKNOWN, and SUSPICIOUS.
EmailIntentType should cover DOCUMENT_INGEST, THREAD_REPLY, APPROVAL_ACTION, CLARIFICATION_RESPONSE, NOTIFICATION, and MANUAL_TRIAGE.
EmailActionType should cover document creation, case creation, case linking, procurement request linking, supplier quotation creation, extraction triggers, quotation prefill triggers, reconciliation triggers, procurement analysis triggers, outbound send, reopen action, and ignore.
6. Domain routing architecture
A deterministic-first EmailRoutingService should decide where each message belongs. Inputs include mailbox configuration, headers, thread context, body preview, attachment metadata, linked participants, known entities, subject/body references, and sender trust level.
Routing rules should run in priority order. Group 1: explicit mailbox routing. Group 2: thread continuation routing. Group 3: attachment-type routing. Group 4: subject/body entity routing. Group 5: sender-role routing. Group 6: fallback LLM assist only when deterministic confidence is below threshold.
7. Domain handler architecture
Base contract: BaseEmailDomainHandler with can_handle() and handle() methods.
APEmailHandler supports new AP invoice intake, supporting document receipt, vendor clarification replies, reviewer or manager actions, and thread attachment linking to existing AP cases. For a new AP invoice, it should validate sender and attachment, create DocumentUpload, mark source channel as EMAIL, trigger extraction, and continue the standard AP flow. For an existing AP case reply, it should link the thread, store attachments as evidence or new DocumentUpload records, and optionally rerun reconciliation or supervisor analysis.
ProcurementEmailHandler supports quotation and proposal intake, BOQ or commercial attachments, clarification replies, quote revisions, follow-up communication for missing commercial terms, and notifications for recommendation or benchmark outcomes. For a new procurement quotation, it should create DocumentUpload, create or link SupplierQuotation, optionally create or link ProcurementRequest, set source to EMAIL, trigger quotation prefill, and later trigger validation, recommendation, or benchmarking. For a clarification reply, it should link the thread to SupplierQuotation or ProcurementRequest, attach revised documents, update structured context, and rerun validation or analysis where needed.
NotificationOnlyHandler covers informational emails that do not mutate domain state. ManualTriageHandler creates a triage item and assigns it to the configured role when no safe automated route exists.
8. Shared ingestion pipeline
Sequence: webhook or poller -> provider adapter fetches message -> InboundIngestionService normalizes the message -> create EmailThread, EmailMessage, and EmailAttachment rows -> AttachmentService stores files in Blob -> ClassificationService assigns message class -> ThreadLinkingService checks existing links -> EntityLinkingService resolves AP or procurement references -> EmailRoutingService decides the domain handler -> handler executes -> EmailAction, AuditEvent, and ProcessingLog are written.
MailboxService registers subscriptions and manages sync state. InboundIngestionService performs idempotent message ingest and deduplicates by provider message ID and internet message ID. AttachmentService downloads, validates, scans, stores, and hashes attachments. ClassificationService does deterministic classification with optional LLM fallback. ThreadLinkingService maintains thread relationships. EntityLinkingService resolves case, invoice, PO, request, quotation, or run references. OutboundEmailService sends audited, template-driven emails.
9. Provider integration design
Prioritize Microsoft Graph first, with Gmail as a secondary adapter. This is the best fit for enterprise deployments and the broader Microsoft-aligned ecosystem already surrounding the application.
Define a BaseEmailProviderAdapter contract with subscribe_mailbox(), poll_messages(), get_message(), get_attachments(), and send_message(). Use webhook subscriptions as the primary mode, Celery polling as fallback, and backfill/recovery sync by cursor or date window.
10. Celery task architecture
Tasks in apps.email_integration.tasks.py should include sync_mailbox_task, ingest_provider_message_task, process_email_message_task, download_email_attachment_task, route_email_message_task, handle_ap_email_task, handle_procurement_email_task, send_templated_email_task, send_clarification_email_task, relink_email_threads_task, retry_failed_email_actions_task, and mailbox_health_check_task.
Recommended queues: email_ingestion, email_processing, email_outbound, and email_recovery.
11. Changes required in AP domain
Extend AP source channel values to include EMAIL.
Add optional relationships such as APCase.primary_email_thread, APCase.last_email_message, DocumentUpload.source_message, and optionally ReviewAssignment.notification_thread.
Allow email messages and attachments to surface as AP case artifacts or evidence.
After AP email intake, reuse the existing AP logic by triggering extraction, case orchestration, reconciliation reruns, or supervisor reruns when new evidence arrives.
12. Changes required in procurement domain
Add source_channel to SupplierQuotation and optionally to ProcurementRequest if requests can originate directly from email.
Add optional linkage fields such as SupplierQuotation.primary_email_thread, ProcurementRequest.primary_email_thread, and AnalysisRun.trigger_source with values like UI, API, EMAIL, and SYSTEM.
Email handling must plug into the quotation prefill pipeline, clarification response handling, validation reruns, recommendation reruns, benchmark reruns, and supplier follow-up notifications.
13. Agent integration design
Do not give agents raw mailbox access. Use permissioned tools only.
Shared tools should include email_thread_lookup, email_attachment_list, email_body_summary, match_email_to_entity, send_templated_email, and extract_email_intent.
AP wrappers should include attach_email_to_case, extract_case_approval_from_email, and send_vendor_clarification_email.
Procurement wrappers should include attach_email_to_procurement_request, attach_email_to_supplier_quotation, extract_supplier_response_fields, and send_supplier_clarification_email.
On the AP side, these tools can be exposed to SupervisorAgent and other AP agents. On the procurement side, use procurement-specific service wrappers first and do not force immediate convergence into the AP shared AgentRun model.
14. System agent design
Shared system services or system agents should include SYSTEM_EMAIL_INTAKE, SYSTEM_EMAIL_ROUTER, SYSTEM_EMAIL_THREAD_LINKER, and SYSTEM_NOTIFICATION_DISPATCH.
Domain adapters should include SYSTEM_AP_EMAIL_HANDLER and SYSTEM_PROCUREMENT_EMAIL_HANDLER. AP can align these with the existing system-agent model. Procurement should still terminate into procurement services, AnalysisRun, and the ProcurementAgentOrchestrator bridge rather than AP case orchestration.
15. Security, RBAC, and governance
Add platform permissions such as email.view, email.read_thread, email.read_attachment, email.send, email.manage_mailboxes, email.route, and email.triage. Add AP-specific permissions for linking AP cases, sending vendor clarification, and extracting approval intent. Add procurement-specific permissions for linking procurement requests, linking supplier quotations, and sending supplier clarification.
Trust and safety rules must enforce configured mailboxes only, tenant-scoped credentials only, sender trust scoring, blocked auto-approval for unknown senders, mandatory attachment safety checks, and tokenized or validated action codes for approval-by-email.
Every email-triggered mutation should create EmailAction, AuditEvent, ProcessingLog, and where appropriate a decision or routing record. Add trace_id and span_id across ingestion, routing, handling, outbound send, and downstream AP or procurement tasks.
16. API design
Mailbox admin endpoints: POST and GET mailbox endpoints plus sync and test actions.
Email operations endpoints: list and read messages, read threads, route a message, and reprocess a message.
Linking endpoints: link a message to an AP case, procurement request, or supplier quotation.
Outbound endpoints: send a template, send clarification, or send notifications.
Webhook endpoints: Microsoft and Gmail inbound webhook receivers.
17. UI design
Platform pages should include mailbox configuration, email inbox or processing dashboard, failed email actions, triage queue, and email health or sync status.
AP UI additions should include case-level email timeline, linked email thread panel, related attachments from email, and a send clarification action.
Procurement UI additions should include procurement request email timeline, supplier quotation thread panel, clarification request and reply history, and outbound template actions for suppliers.
18. End-to-end flow examples
AP invoice email: vendor sends invoice email -> mailbox webhook fires -> EmailMessage and EmailAttachment created -> routing = AP_INVOICE -> APEmailHandler creates DocumentUpload -> extraction runs -> Invoice created -> APCase created -> reconciliation runs -> agents or supervisor run if needed.
Procurement quotation email: supplier sends quotation or proposal email -> mailbox webhook fires -> EmailMessage and EmailAttachment created -> routing = PROCUREMENT_QUOTATION -> ProcurementEmailHandler creates DocumentUpload -> SupplierQuotation created or linked -> quotation prefill task runs -> user reviews extracted fields -> AnalysisRun created -> validation, recommendation, or benchmark runs.
Procurement clarification reply: system sends supplier clarification email -> supplier replies on same thread -> EmailThread linked to SupplierQuotation or ProcurementRequest -> reply body parsed -> revised attachment stored as DocumentUpload -> quotation prefill or validation rerun triggered -> procurement-side execution records continue to capture analysis-side agent activity.
19. Recommended implementation phases
Phase 1: shared inbound email foundation. Build mailbox config, webhook and polling, EmailMessage/EmailAttachment/EmailThread, attachment-to-blob, deterministic classification, routing decision storage, and manual triage queue.
Phase 2: AP and procurement handlers. Build APEmailHandler and ProcurementEmailHandler, create DocumentUpload from attachments, extend source channels, and establish thread/entity linking.
Phase 3: outbound communication. Build EmailTemplate, OutboundEmailService, AP clarification emails, procurement supplier clarification emails, and notification sends.
Phase 4: agent-safe email tools. Build shared tools, AP wrappers, procurement wrappers, and permission integration.
Phase 5: advanced actions. Build approval-by-email with tokenization, email analytics, SLA monitoring, suggested replies or summarization, and auto-reopen logic based on reply content.
20. Recommended target-state decisions
Decision 1: Use one shared apps.email_integration app for all email.
Decision 2: Use domain handlers for AP and procurement instead of one generic business processor.
Decision 3: Keep DocumentUpload as the common attachment ingestion backbone across both domains.
Decision 4: Use deterministic routing first and LLM only when confidence is low.
Decision 5: Do not give agents unrestricted mailbox access; use scoped and permissioned tools.
Decision 6: Keep AP and procurement execution models separate after routing. AP continues through case and reconciliation orchestration. Procurement continues through request, quotation, and analysis orchestration.
21. Final architecture summary
A tenant-scoped shared email integration layer ingests, normalizes, classifies, links, and routes emails into either AP or procurement domain handlers, uses DocumentUpload as the common attachment backbone, and preserves each domain’s existing orchestration, RBAC, audit, and async execution model.

Appendix C. Source alignment notes
This target architecture is intentionally aligned to the inspected platform design: AP is document and case driven; procurement uses ProcurementRequest, SupplierQuotation, AnalysisRun, and ProcurementAgentOrchestrator; DocumentUpload is the shared backbone; and the application already relies on tenant scope, Celery-based async execution, and governance-oriented audit and RBAC patterns.
