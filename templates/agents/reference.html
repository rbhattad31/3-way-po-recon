End-to-End Processing Flow

Every invoice creates an AP Case driven through a sequence of stages by the Case Orchestrator. The extraction pipeline uses a two-agent architecture (Extraction Agent + Understanding Agent for low confidence). The processing path (2-Way, 3-Way, or Non-PO) is resolved automatically and determines which stages run. LLM-powered agents use a ReAct loop (Reason → Act → Observe) with up to {{ max_tool_rounds }} tool-calling rounds.

When do agents run?
  • Extraction (always) — Invoice Extraction agent extracts structured data from OCR output using GPT-4o with JSON mode.
  • Extraction (low confidence) — Invoice Understanding agent validates and cross-checks when confidence < {{ confidence_threshold }}%.
  • PO Retrieval — PO Retrieval agent is the fallback when deterministic lookup fails.
  • GRN Analysis — GRN Retrieval agent investigates receipt discrepancies (3-Way only, conditional).
  • Exception Analysis — Exception Analysis + Reconciliation Assist agents reason over exceptions.
  • Review Routing — Review Routing agent assigns the case to the right reviewer queue.
  • Case Summary — Case Summary agent produces the final narrative for reviewers.

How Agents Choose a Recommendation
  • Each agent receives the reconciliation exceptions and uses tools to gather context.
  • The agent reasons through the exceptions and picks the most appropriate recommendation type.
  • A confidence score (0–100%) indicates how certain the agent is about its recommendation.
  • If confidence is below the threshold, the case is escalated for human review.
  • Multiple agents may run in sequence (e.g., Exception Analysis → Review Routing → Case Summary).
  • The Policy Engine may auto-close cases that fall within auto-close tolerance bands (qty: 5%, price: 3%, amount: 3%).

RBAC Middleware Chain
  1. RequestTraceMiddleware — Creates root TraceContext, enriches with RBAC snapshot (actor_user_id, actor_email, actor_primary_role, actor_roles_snapshot). Sets X-Trace-ID / X-Request-ID response headers.
  2. RBACMiddleware — Pre-loads user.get_role_codes() and user.get_effective_permissions() on every authenticated request. Cached on the user instance for the request lifecycle.
  3. LoginRequiredMiddleware — Redirects all anonymous requests to login (except /admin/, /accounts/, /api/).
  4. rbac_context (context processor) — Injects user_permissions, user_role_codes, and is_admin into every template context.
